# `/supi:qa` monorepo plan

## Goal

Make `/supi:qa` operate on one selected app/package at a time instead of assuming one app type, one dev command, one port, and one session/config namespace per repo root.

## Current monorepo-sensitive assumptions

Observed in:
- `src/commands/qa.ts`
- `src/qa/detect-app-type.ts`
- `src/qa/discover-routes.ts`
- `src/qa/config.ts`
- `src/storage/qa-sessions.ts`

Current behavior assumes:
- one app per cwd
- one detected framework per cwd
- one `e2e-qa.json` per project root
- one QA session namespace per project root
- route discovery relative to one root directory tree

## Design direction

### Invocation model

Use one target per invocation.

Behavior:
- allow `--target <package>`
- otherwise pick from detected app-like packages with changed packages first
- once a package is selected, all app detection and route discovery happen relative to `target.packageDir`

### Config and session model

Per-package QA runs should use per-package config/session namespacing so multiple frontend apps in one monorepo do not overwrite each other.

### Detection model

`detectAppType` and `discoverRoutes` should accept a package root rather than assuming repo root.

## Dependencies on shared foundation

Required before implementation:
- shared `WorkspaceTarget`
- target picker / `--target` helper
- package manifest helper
- package-scoped state/config path builder
- optional root/workspace config layering if QA defaults should be inherited from root config

## Suggested parallel workstreams

### Agent A — Command flow and package selection

Files:
- `src/commands/qa.ts`
- QA command tests

Scope:
- add target selection before QA setup/run
- thread selected target through the existing QA orchestration path

### Agent B — App detection and route discovery

Files:
- `src/qa/detect-app-type.ts`
- `src/qa/discover-routes.ts`
- QA detection/discovery tests

Scope:
- make detection operate on `target.packageDir`
- make route discovery operate on `target.packageDir`
- preserve current framework support

### Agent C — Config and session namespacing

Files:
- `src/qa/config.ts`
- `src/storage/qa-sessions.ts`
- session/config tests

Scope:
- namespace config per package
- namespace QA sessions per package
- surface package context in session summaries

## Acceptance criteria

- `/supi:qa --target <package>` runs against only the selected app/package
- app detection and route discovery are rooted at the selected package directory
- QA config and sessions for one package do not collide with another package
- classic single-app repos keep current behavior

## Risks

- wrong package selection will silently produce the wrong framework/route set unless the selected package is shown clearly in the UI
- per-package ports and base URLs must be explicit; a single repo-wide port model will fail in real monorepos
- package detection for non-web packages should fail fast instead of offering unusable QA flows

## Explicit non-goals for this wave

- multi-app QA in one invocation
- monorepo-wide route aggregation
- orchestration across several frontend packages in one session
